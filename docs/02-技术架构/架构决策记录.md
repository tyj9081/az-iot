## ADR-01 — 读数表按月分区

**状态**: Accepted

**Context**: dev_device_reading 按 30 秒采集周期，90 天可达 2.4 亿行。

**Decision**: 使用 MySQL RANGE PARTITION BY MONTH(read_at)，按月自动创建 + DROP 清理。

**Consequences**: 主键从 `(id)` 变为 `(id, read_at)` 联合主键。查询必须带 read_at 条件才能享受分区剪枝。

## ADR-02 — 数据保留两层

**状态**: Accepted

**Decision**:

| 层 | 存储 | 默认保留 | 配置 | 清理方式 |
|---|------|---------|------|---------|
| 采集端 | SQLite | 7 天覆盖 | config.toml | 定时 DELETE |
| 服务端 | MySQL 分区表 | 90 天 | sys_config API | DROP 分区 |
| 审计日志 | MySQL | 180 天 | sys_config API | 批量 DELETE |

**Consequences**: 两端独立管理，各自关注不同生命周期。

## ADR-03 — 双通道上报

**状态**: Accepted

**Decision**:

```
通道A (实时): latest (QoS0) → 服务端内存 → WebSocket → 前端 (<100ms)
通道B (历史): 降采样聚合 → reading (QoS1) → MySQL 分区表 → API
```

- 通道A 不进 DB，纯内存缓存 + WebSocket 广播
- 通道B 采集端本地 Aggregator 攒够窗口 → avg/max/min → 上报

**Consequences**: 实时感知和长期存储解耦。前端趋势图用 `avg` 画线，`max/min` 画误差带。秒级波形查看需直接访问采集端 SQLite。

## ADR-04 — 告警边缘判断

**状态**: Accepted

**Decision**: 采集端每次 poll 后本地阈值判断 (非服务端远程判断)。断网状态下 DO 继电器仍可联动声光报警。服务端只负责告警记录存储 + 通知分发。

**Consequences**: 告警延迟从 "采集→MQTT→服务端→判断" 缩短为 "采集→本地判断→MQTT alarm"。断网不盲。

## ADR-05 — 采集端降采样

**状态**: Accepted

| 采集周期 | 窗口 | MySQL 日写入 (47台×20点) |
|---------|------|------------------------|
| 1秒 | 60秒 | 27万 |
| 5秒 | 180秒 | 9万 |
| 10秒 | 300秒 | 5.4万 |
| ≥30秒 | 不聚合 | ≤28万 |

上报荷载: `{sensor_code, avg, max, min, sample_count, window_start, window_end}`

**Consequences**: MySQL 日写入量不随采集频率增长，始终 ≤28 万条。代价是前端看不到 1 秒级波形细节。

## ADR-06 — 模块化单体架构

**状态**: Accepted

**Decision**: Maven 技术层切模块(7 个), Java 包按业务域分包(device/collector/system)。不做微服务拆分。

**Consequences**: 部署简单，调试无网络开销。未来某域需要独立扩容时再拆分。

## ADR-07 — MQTT 三层安全模型

**状态**: Accepted | 2026-06-26

**Context**: 服务端和采集端通过 MQTT 传输设备读数和控制指令，当前设计未涉及安全措施。裸 TCP + 无认证 = 任何人可嗅探/注入数据。

**Decision**: 三层安全体系:

### 传输层 — TLS 1.2+

```
采集端/TLS → EMQX(TLS) ← TLS/服务端
```
- EMQX 开启 TLS 监听 (端口 8883)
- 采集端和服务端必须通过 TLS 连接
- 证书管理: 内网部署用自签 CA → 签发客户端证书; 公网部署用 Let's Encrypt

### 认证层 — 双因子

| 因子 | 用途 | 配置位置 |
|------|------|---------|
| 用户名+密码 | 第一因子，EMQX HTTP Auth 回调服务端验证 | dev_collector.mqtt_username / mqtt_password (服务端生成) |
| 客户端证书 (可选增强) | 第二因子，TLS 双向认证 | 采集端 config.toml 指定证书路径 |

- 每个采集器独立凭证，注册时服务端自动生成 → 下发到采集端 config.toml
- 凭证存储在 `dev_collector` 表的 `mqtt_username` + `mqtt_password_hash` 字段 (password 哈希存储，不存明文)
- 服务端使用独立的管理凭证连接 EMQX (不在采集器表内)

### 授权层 — EMQX ACL

基于 client_id 的 Topic ACL 规则:

| 角色 | 允许 PUB | 允许 SUB |
|------|---------|---------|
| 采集器 (client_id=xxx) | `neuron/xxx/reading`, `neuron/xxx/latest`, `neuron/xxx/heartbeat`, `neuron/xxx/alarm`, `neuron/xxx/config/ack`, `neuron/xxx/status` | `neuron/xxx/config/delta`, `neuron/xxx/config/full`, `neuron/xxx/cmd` |
| 服务端 | `neuron/+/config/delta`, `neuron/+/config/full`, `neuron/+/cmd` | `neuron/+/reading`, `neuron/+/latest`, `neuron/+/heartbeat`, `neuron/+/alarm`, `neuron/+/config/ack`, `neuron/+/status` |
| **禁止** | — | `neuron/+/cmd_result` (采集器仅能 PUB, 服务端仅能 SUB) |

- 采集器只能在自己的 namespace `neuron/{own_id}/` 下操作
- **任何采集器都不得 PUB 其他采集器的 topic，也不得 SUB 其他采集器的 config**

### dev_collector 表扩展

新增字段:
- mqtt_username VARCHAR(64) — MQTT 认证用户名
- mqtt_password_hash VARCHAR(256) — BCrypt 哈希后的密码
- mqtt_tls_enabled TINYINT DEFAULT 0 — 是否启用 TLS

注册采集器时，服务端自动生成随机凭证:
```java
String username = "collector_" + UUID.randomUUID().toString().substring(0, 8);
String password = UUID.randomUUID().toString().substring(0, 16);
// 存储 password_hash = BCrypt.encode(password)
// 下发到采集端 config.toml 的 mqtt.username/mqtt.password
```

**Consequences**:
- 采集端首次部署需要手动配置 username/password (无法远程下发——鸡生蛋问题)
- 解决方案: 运维人员在 Web 界面创建采集器 → 复制生成的凭证 → SSH 到 BC-U101 粘贴至 config.toml
- TLS 证书管理增加运维复杂度——内网环境下可先跳过 TLS (通过物理隔离保障)，公网部署时强制启用
- 冒名上报风险: 即使知道 client_id，没有对应凭证也无法连接 EMQX

## ADR-08 — 协议枚举前后端编译期对齐

**状态**: Accepted | 2026-06-26

**Context**: 原有设计中采集端(Rust)定义了 4 种协议枚举且全部使用 Mock 驱动，服务端(Java)协议定义靠数据库 `dev_protocol` 表，两端无编译期一致性保证。数据库临时改一行 `code` 即可导致采集端无法识别协议。

**Decision**:

1. **采集端扩充 ProtocolType 枚举至 18 种工业协议**，每种 `#[serde(rename)]` 输出固定 `code` 字符串
2. **服务端新增 `com.aziot.common.enums.ProtocolType` 枚举**，18 个常量，`code` 字段与 Rust serde rename 严格逐字一致
3. **数据库 `dev_protocol` 表**通过 Flyway `V1__init_schema.sql` 初始化，`code` 列值与两端枚举对齐
4. **禁止 Mock**: 每个协议必须有真实的帧级驱动实现，DriverFactory 中不再包含 `MockModbusDriver`
5. **平台适配**: CAN Bus 仅在 Linux 可用 (`#[cfg(target_os)]`)，其他平台编译通过但运行时明确报错；PROFIBUS 同理

**协议清单**:

| 类别 | 协议 | Rust code | Java code | 驱动文件 |
|------|------|-----------|-----------|---------|
| 串口(6) | Modbus RTU, DL/T645 2007/1997, IEC 101, CAN Bus, PROFIBUS DP | — | — | 6 文件 |
| TCP/IP(9) | Modbus TCP, IEC 104, DNP3, OPC UA, BACnet/IP, S7, FINS, EtherNet/IP, Mitsubishi MC | — | — | 9 文件 |
| 上层(3) | MQTT, SNMP v2c, HTTP JSON | — | — | 3 文件 |

**Consequences**:
- 新增协议必须同时修改三处: Rust enum + Java enum + Flyway INSERT，编译期即可发现遗漏
- 前端下拉框不再硬编码 `pageSize: 999` — 协议列表只需调 `GET /protocols` 获取数据库现有记录
- `DevProtocolService` 创建/更新时可通过 `ProtocolType.isValidCode()` 校验合法性
