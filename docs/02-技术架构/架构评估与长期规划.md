> 架构师: Jeff | 日期: 2026-06-26 | 版本: v1.0

## 一、执行摘要

AZ-IOT 是一个设计规范的**三层 IoT 能源管理平台**，面向工业园区/楼宇场景。项目已完成 MVP 阶段主要开发工作，整体完成度约 **75%**。架构决策（8 项 ADR）质量高，数据分层策略、双通道上报、边侧告警判断等设计在同类项目中属于上乘。

**当前核心缺口**在 Rust 采集端——调度主循环、MQTT 配置同步、SQLite 本地缓存、遥测心跳四个模块尚未实现。此外全链路集成测试、CI/CD 流水线为空白。

本报告从五个维度做全面评估：项目完成度、架构质量、缺口分析、长期演进路线、实施优先级建议。

## 三、架构质量评估

### 3.1 架构模式选择

**当前决策: 模块化单体 (ADR-06)**

| 评价维度 | 评分 | 说明 |
|----------|------|------|
| 合理性 | ⭐⭐⭐⭐⭐ | 小团队 + MVP 阶段，模块化单体是最佳选择 |
| 模块边界 | ⭐⭐⭐⭐ | Maven 7 模块 + 按业务域分包，边界清晰 |
| 扩展性 | ⭐⭐⭐ | 当前需要独立扩容时需拆分，但ADR预留了路径 |
| 复杂度 | ⭐⭐⭐⭐ | 无分布式问题，部署调试简单 |

**评价**: 这是项目最正确的架构决策之一。Maven 多模块 + 按域分包的方式，既保持了单体的部署便利性，又通过编译期模块隔离防止了代码腐化。未来某域需要独立扩容时（如读数写入量暴涨），可以针对性拆分。

### 3.2 数据架构

| ADR | 决策 | 评价 |
|-----|------|------|
| ADR-01 按月分区 | MySQL RANGE PARTITION BY MONTH | ⭐⭐⭐⭐⭐ 经典方案，2.4亿行90天无忧 |
| ADR-02 两层保留 | SQLite 7天 + MySQL 90天 + 审计180天 | ⭐⭐⭐⭐ 合理，但SQLite循环覆盖策略待验证 |
| ADR-03 双通道上报 | 通道A(实时QoS0) + 通道B(降采样QoS1) | ⭐⭐⭐⭐⭐ 亮点设计，实时/历史解耦 |
| ADR-05 采集端降采样 | 窗口聚合，日写入≤28万 | ⭐⭐⭐⭐ 有效控制DB开销 |

**评价**: 数据架构设计是项目第二大亮点。双通道上报同时满足了前端实时刷新（毫秒级）和历史数据查询（天/周/月粒度的趋势图），且通过采集端降采样保证了 MySQL 写入量可控。AD-01 的分区策略在 2.4 亿行规模下非常必要。

### 3.3 安全架构

| 层面 | 实现 | 评价 |
|------|------|------|
| 传输安全 | JWT Bearer Token (30min) + Refresh Token (7d) | ⭐⭐⭐⭐ 标准方案 |
| 密码存储 | BCrypt 哈希 | ⭐⭐⭐⭐⭐ |
| 防枚举 | 统一"用户名或密码错误"提示 | ⭐⭐⭐⭐ |
| 限流 | IP 每分钟≤5次登录，超限429 | ⭐⭐⭐⭐ |
| 权限 | RBAC 三角色(admin/operator/viewer) + 树形权限 | ⭐⭐⭐⭐ |
| 审计 | @AuditLog AOP 自动记录操作 | ⭐⭐⭐⭐ |
| MQTT安全 | ADR-07 TLS1.2+双因子+ACL | ⭐⭐⭐⭐⭐ 设计完整 |

**待加固**: Refresh Token 当前是 JWT 无状态方式，不推荐用于需要主动撤销的场景。建议升级为 Redis 白名单（已在 P1 待修复中）。

### 3.4 关键架构风险

| 风险 | 严重程度 | 说明 |
|------|----------|------|
| 调度主循环未实现 | 🔴 高 | Collector::run() 仅打日志，全链路无法实际跑通 |
| 三 stub 模块空 | 🔴 高 | 无配置同步=设备上线后采集端无法感知；无本地缓存=断网丢数据；无遥测=运维盲盒 |
| 全链路集成测试为零 | 🔴 高 | 无法保证三端数据通路正确性 |
| 无 CI/CD | 🟠 中 | 回归全靠人工，风险高 |
| 前后端零测试 | 🟠 中 | 重构时无安全网 |
| MQTT重连未实现 | 🟡 低 | 已设计但未编码，EMQX 重启后采集端会丢失连接 |

## 六、架构决策补充建议

### ADR-09 (建议) — 全链路测试策略

**Context**: 当前 Rust 侧 9 个单元测试，前后端零测试。三端数据通路无自动化验证。

**Decision**: 建立三层测试金字塔:

```
        E2E (Cypress/Playwright)
       ┌──────────────────────┐
      │    全链路场景验证      │
     │    (3-5个核心场景)       │
    └──────────────────────┘
         集成测试 (SpringBootTest + H2)
       ┌──────────────────────┐
      │   API → DB 完整链路     │
     │   (每 Controller 2-3个) │
    └──────────────────────┘
        单元测试 (JUnit / Vitest / cargo test)
       ┌──────────────────────┐
      │   业务逻辑 + 协议解析    │
     │   (关键路径全覆盖)       │
    └──────────────────────┘
```

**Consequences**: 开发时间增加 20-30%，但重构安全网建立，长期维护成本下降 50%+。

### ADR-10 (建议) — 引入 CI/CD 流水线

**Context**: 无 CI 配置，提交后无自动验证。

**Decision**: GitHub Actions 三端并行流水线:

```yaml
# .github/workflows/ci.yml
jobs:
  java-build:
    runs-on: ubuntu-latest
    steps: [checkout, java 17, mvn verify]
  
  rust-build:
    runs-on: ubuntu-latest
    steps: [checkout, cargo test, cargo clippy, cargo build]
  
  web-build:
    runs-on: ubuntu-latest
    steps: [checkout, pnpm install, pnpm build, pnpm lint]
```

**Consequences**: 每次 PR 自动验证，杜绝"在我机器上能跑"问题。

### ADR-11 (建议) — 协议驱动可插拔化

**Context**: 当前 18 个协议驱动全部编译进 binary。如果某部署场景只需 Modbus + DL645，其余 16 个驱动白白增加 binary 体积和攻击面。

**Decision**: 利用 Rust feature flag 实现编译时裁剪:

```toml
[features]
default = ["modbus", "dlt645"]
modbus = []
dlt645 = []
iec104 = []
# ...
all = ["modbus", "dlt645", "iec104", ...]
```

**Consequences**: Binary 体积可控，最小部署仅 Modbus 一种驱动。代价是维护 feature 组合矩阵。

## 八、总结

### 优势

1. **架构决策质量高** — 8 项 ADR 均有明确上下文和 trade-off 分析，非盲目选型
2. **数据架构设计优秀** — 双通道上报 + 按月分区 + 边侧降采样，工业 IoT 场景的经典解
3. **协议驱动层扎实** — 18 种协议真实帧级实现，非 Mock，非空壳
4. **文档体系完善** — 12 份文档覆盖产品/架构/规范/API，团队新人可快速上手
5. **安全设计完整** — JWT+BCrypt+RBAC+审计+限流+MQTT三层安全，无硬编码漏洞

### 劣势

1. ~~采集端核心逻辑缺失~~ → **已解决 (2026-06-26)**
2. **全链路验证为零** — 无法确认数据能否从 BC-U101 正确流到前端页面
3. **无自动化测试** — 任何代码变更无安全网，重构风险极高
4. **无 CI/CD** — 回归全靠人工
