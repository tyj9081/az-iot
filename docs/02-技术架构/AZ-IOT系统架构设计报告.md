## 1. 系统概述

### 1.1 项目定位

AZ-IOT 是一套面向工业园区与智能楼宇的**能源管理 IoT 平台**，以嵌入式一体化监控主机 BC-U101 为边缘采集枢纽，构建从现场设备数据采集、服务端存储加工、到前端可视化展示的完整数据链路。

### 1.2 核心价值链

```
现场设备 → BC-U101 采集器 → Rust 采集端 → MQTT → Java 服务端 → Vue3 前端
  (仪表/传感器)    (MODBUS/645/串口)    (协议解析)  (消息总线)  (存储/逻辑)   (可视化)
```

### 1.3 功能范围 (MVP)

| 模块 | 能力 |
|------|------|
| 设备模板管理 | 品牌/型号/协议字典配置，寄存器点表 CRUD + Excel 批量导入 |
| 采集器管理 | BC-U101 注册，串口自动生成 (COM1-COM10)，MQTT 凭证签发 |
| 设备纳管 | 绑定型号/串口/从站地址，配置变更自动 MQTT 下发至采集端 |
| 数据采集 | 18 种工业协议解析，双通道上报 (实时通道 + 历史通道) |
| 数据展示 | Dashboard 概览，设备详情 (点表驱动通用渲染器)，历史趋势图 |
| 告警管理 | 边缘判断 (采集端本地阈值判定)，断网仍可联动 DO 继电器报警 |
| 系统管理 | JWT 双 Token 认证，三角色 RBAC，操作审计日志，MQTT 三层安全 |

### 1.4 关键指标

| 指标 | 目标值 | 设计依据 |
|------|--------|---------|
| 单采集端设备数 | ≤ 50 台 | BC-U101 串口 + 扩展卡上限 |
| 数据采集时延 | < 1s | 双通道实时上报 (QoS0) |
| MySQL 日写入量 | ≤ 28 万条 | 采集端降采样聚合 |
| 历史数据保留 | 90 天 (MySQL) + 7 天 (SQLite) | 可配置 |
| 历史数据规模 | ≤ 2.4 亿行 (90 天) | RANGE 分区承载 |
| 告警响应延迟 | < 100ms | 本地判断，不跨网络 |
| 单 API 响应 | P99 < 500ms | — |

### 2.1 限界上下文 (Bounded Context)

采用 DDD 战术设计划分三个核心限界上下文：

```
┌──────────────────────┐   ┌──────────────────────┐   ┌──────────────────────┐
│  设备模板上下文        │   │  采集管理上下文        │   │  系统管理上下文        │
│  Device Template BC   │   │  Collection BC        │   │  System BC           │
├──────────────────────┤   ├──────────────────────┤   ├──────────────────────┤
│ • 协议字典            │   │ • 采集器注册           │   │ • 用户管理            │
│ • 品牌管理            │   │ • 串口配置             │   │ • 角色权限            │
│ • 设备型号            │   │ • 设备纳管             │   │ • 审计日志            │
│ • 寄存器点表          │   │ • 配置下发             │   │ • 系统配置            │
│                      │   │ • 数据采集             │   │ • 认证授权            │
│                      │   │ • 告警管理             │   │                      │
│                      │   │ • 数据上报             │   │                      │
└──────────────────────┘   └──────────────────────┘   └──────────────────────┘
```

### 2.2 上下文映射 (Context Map)

```
设备模板上下文 ──────────────────→ 采集管理上下文
  (上游/Open Host Service)          (下游/Conformist)
  提供: 型号 + 点表查询接口          消费: 创建 Device 时引用 model_id

系统管理上下文 ──────────────────→ 采集管理上下文
  (上游/Open Host Service)          (下游/Conformist)
  提供: 用户认证与权限校验            消费: API 调用身份验证

采集管理上下文 ──────────────────→ 系统管理上下文
  (上游/Open Host Service)          (下游/Conformist)
  提供: 告警事件                     消费: 告警通知分发
```

### 2.3 聚合根 (Aggregate Root)

| 聚合 | 聚合根 | 内部实体/值对象 | 不变量 |
|------|--------|---------------|--------|
| 设备模板 | DeviceModel | Protocol, Manufacturer, RegisterMap[] | 型号关联的协议必须有效；同型号内 sensor_code 唯一 |
| 采集器 | Collector | SerialPort[], Device[] | COM端口名不重复；Device.slave_addr 在同一串口下唯一 |
| 采集数据 | DeviceReading | — | read_at 必须在有效分区范围内 |
| 用户权限 | SysUser | SysRole[], SysPermission[] | 角色不能嵌套超过 2 层 |

### 3.3 架构模式

| 决策 | 模式 | 依据 |
|------|------|------|
| 服务端架构 | **模块化单体** (ADR-06) | 小团队 (≤4人), 业务边界清晰, 无需独立扩容 |
| 模块划分 | Maven 7 模块 + 按业务域分包 | 编译期模块隔离 + 运行期统一部署 |
| 采集端架构 | **Actor 模式** (每个设备一个 Task) | 设备间轮询隔离，单设备故障不传染 |
| 数据通道 | **发布-订阅** (MQTT) | 采集端→服务端异步解耦，断线重连自带 |

### 3.4 关键流程

#### 设备纳管流程

```
Web 创建设备 → 服务端入库 → ConfigPushService 构造 ConfigDelta
  → MQTT Publish neuron/{id}/config/delta
  → 采集端 ConfigSync 接 delta → 校验版本 → 注册到 DeviceRegistry
  → Collector 主循环纳入轮询 → MQTT Publish neuron/{id}/config/ack
```

#### 数据采集流程

```
Collector::run() 主循环:
  tick (1s)
  → 遍历 bus_groups (同一串口设备归组轮询)
  → ProtocolDriver::collect(device, data_points)
  → 解析寄存器值 → DataPoint
  ├─ 通道A: latest = LatestReading → RealtimePublisher → MQTT QoS0 → 服务端 WebSocket → 前端
  ├─ 通道B: Aggregator.add(reading) → 窗口满 → AggregatedReading → MQTT QoS1 → MySQL
  └─ 告警: AlarmEngine.check(value) → 超阈值 → DO 继电器联动 + MQTT alarm → 服务端通知
```

## 4. 技术栈选型

### 4.1 总体技术栈

| 层 | 技术 | 版本 | 选型理由 |
|----|------|------|---------|
| **前端** | Vue 3 + TypeScript | 3.4+ | Composition API, 生态成熟 |
| | Element Plus | 2.x | 企业级组件库，中文支持好 |
| | Pinia | 2.x | Vue 3 官方状态管理 |
| | Vite | 5.x | 极速 HMR，ESM 原生 |
| **服务端** | Java + Spring Boot | 17 / 3.2 | 团队主力语言，生态丰富 |
| | MyBatis-Plus | 3.5.5 | 零 SQL CRUD，Lambda 类型安全 |
| | Flyway | — | 数据库版本管理，不可变迁移 |
| | JJWT | 0.12.3 | 轻量 JWT，无 Jackson 依赖冲突 |
| **采集端** | Rust | 1.80+ | 零成本抽象 + 内存安全 + 串口原生支持 |
| | tokio | 1.x | 异步运行时，串口/TCP 统一调度 |
| | rumqttc | 0.24 | MQTT v5，异步，与 tokio 深度集成 |
| **消息** | EMQX | 5.x | 高并发 MQTT Broker，内置 ACL |
| **存储** | MySQL 8.0 | 8.0 | 分区表支持，InnoDB 性能稳定 |
| | SQLite (采集端) | — | 嵌入式，免运维，断网可用 |

### 4.2 技术栈风险矩阵

| 技术 | 风险 | 缓解措施 |
|------|------|---------|
| Rust | 团队学习曲线陡峭 | 架构师主导，代码规范严格 |
| EMQX | 运维复杂度 | 内网部署用社区版，ACL 配置文件化 |
| MySQL 分区 | 不支持外键 | 应用层保证引用完整性 |
| MQTT QoS0 | 可能丢消息 | 仅用于实时通道，历史通道用 QoS1 |

### 5.2 表设计要点

| 表 | 行估算 | 关键设计 |
|----|-------|---------|
| dev_protocol | 18 行 | 预置 18 种协议 code |
| dev_manufacturer | ~100 行 | 电力仪表主流品牌 |
| dev_device_model | ~200 行 | 与 protocol + manufacturer 级联 |
| dev_register_map | ~2000 行 | (model_id, sensor_code) 唯一约束 |
| dev_collector | ~50 行 | mqtt_client_id 唯一，凭证自动生成 |
| dev_serial_port | ~500 行 | (collector_id, port_name) 唯一；bus_param 存 JSON |
| dev_device | ~500 行 | slave_addr 串口内唯一 |
| dev_device_reading | **2.4 亿行** (90天) | **RANGE 按月分区**，3 个复合索引 |
| dev_device_alarm_config | ~500 行 | (device_id, sensor_code) 唯一 |

### 5.3 双通道数据策略

```
采集端 poll 每 tick:
┌─────────────┐
│ raw_value   │
└──────┬──────┘
       ├── 通道A (实时) ──────────────────────────────────────
       │   LatestReading {sensor_code, value, quality, ts}
       │   → MQTT QoS0, topic: neuron/{id}/latest
       │   → 服务端内存 Map (ConcurrentHashMap, 无持久化)
       │   → WebSocket 广播 → 前端实时刷新 (<100ms)
       │   → **不进 DB**
       │
       └── 通道B (历史) ──────────────────────────────────────
           Aggregator.append(raw_value)
           → 窗口满 (60s~300s, 按采集周期)
           → AggregatedReading {sensor_code, avg, max, min, count, window_start, window_end}
           → MQTT QoS1, topic: neuron/{id}/reading
           → 服务端批量 INSERT → MySQL 分区表
           → API 查询 → 前端趋势图 (avg 画线, max/min 画误差带)
```

### 5.4 索引策略

```sql
-- dev_device_reading 核心索引 (分区表内)
idx_device_time      (device_id, read_at)        -- 设备详情页按时间查询
idx_register_time    (register_id, read_at)       -- 单采集点趋势图
idx_device_register  (device_id, register_id, read_at)  -- 设备+采集点联合过滤
```

## 6. 安全架构

### 6.1 安全体系全景

```
┌─────────────────────────────────────────────────────┐
│                    安全体系                           │
├───────────────┬───────────────┬─────────────────────┤
│  传输安全      │  认证鉴权      │  数据安全            │
├───────────────┼───────────────┼─────────────────────┤
│ Web: HTTPS    │ JWT 双Token   │ 密码 BCrypt 哈希     │
│ MQTT: TLS 1.2 │ RBAC 三角色    │ 凭证不存明文         │
│   (可选)      │ 登录限流       │ 审计日志不可篡改     │
│               │ MQTT 双因子    │                     │
└───────────────┴───────────────┴─────────────────────┘
```

### 6.2 认证流程

```
用户登录:
  POST /api/v1/auth/login {username, password}
  → BCrypt.verify(password, hash)
  → 检查账号状态 (enabled/locked)
  → 检查登录限流 (IP 每分钟 ≤5次)
  → 生成 accessToken (30min, JWT) + refreshToken (7d, JWT)
  → 返回前端

API 请求:
  Authorization: Bearer <accessToken>
  → JwtAuthenticationFilter 拦截
  → 验证签名 + 过期时间
  → 设置 SecurityContext (username + authorities)
  → 放行

Token 刷新:
  POST /api/v1/auth/refresh {refreshToken}
  → 验证 refreshToken 有效
  → 生成新的 accessToken + refreshToken
  → 返回前端 (旧 refreshToken 加入黑名单)
```

### 6.3 RBAC 权限模型

```
┌──────────┐    N:M     ┌──────────┐    N:M     ┌──────────────┐
│ SysUser  │◄──────────▶│ SysRole  │◄──────────▶│ SysPermission│
│          │ user_role   │          │ role_perm  │ (树形结构)    │
└──────────┘            └──────────┘            └──────────────┘

预置角色:
  admin    — 全部权限
  operator — 设备管理 + 数据查看 + 告警处理
  viewer   — 仅数据查看 (只读)

权限类型:
  menu   — 菜单可见性
  button — 操作按钮 (新增/编辑/删除)
  api    — 后端 API 访问 (颗粒度控制)
```

### 6.4 MQTT 三层安全 (ADR-07)

| 层 | 机制 | 说明 |
|----|------|------|
| **传输层** | TLS 1.2+ (可选) | EMQX 端口 8883，内网可跳过 |
| **认证层** | 用户名+密码 / 客户端证书 | 每台采集器独立凭证，服务端注册时自动生成 |
| **授权层** | EMQX ACL | 采集器仅能 PUB/SUB 自己的 `neuron/{own_id}/` namespace |
