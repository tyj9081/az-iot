## 一、项目总体概况

| 维度 | 数据 |
|------|------|
| **项目定位** | 工业园区 / 楼宇能源管理 IoT 平台 |
| **架构模式** | 三层：前端 (Vue3) → 服务端 (Spring Boot) → 采集端 (Rust) → BC-U101 → 现场设备 |
| **协议覆盖** | Modbus RTU/TCP、DL/T645、BACnet、SNMP、MQTT、OPC UA、IEC 103/104、DNP3、S7、FINS、EtherNet/IP、Mitsubishi MC、HostLink、CAN Bus、PROFIBUS、KNXnet/IP — **共 18 种** |

| **核心矛盾** | 设计架构 A 级 vs 代码实现 C+ 级。文档描述的三通道高速公路，代码里只跑了一条单车道 |

## 二、设计亮点

以下是项目架构设计层面做得出色的部分，**值得保留和强化**：

| # | 亮点 | 说明 |
|---|------|------|
| 1 | **18 种工业协议全部帧级驱动实现** | 全行业极少见的全栈协议覆盖，全部是真实编码驱动（非 Mock） |
| 2 | **双通道上报架构** | 实时通道 (QoS0，不落库) 与历史聚合通道 (QoS1，落库) 分离，各司其职 |
| 3 | **边缘告警判断** | 采集端本地阈值判断，断网仍可联动 DO 继电器 — 工业场景正确决策 |
| 4 | **降采样策略** | 1s → 60s 窗口，5s → 180s 窗口，>=30s 不聚合，有效控制 MySQL 写入压力 |
| 5 | **按月分区存储** | 时序数据最佳实践，支持按保留天数自动清理 |
| 6 | **MQTT 三层安全模型** | 传输层 (TLS) + 认证层 (独立凭证 BCrypt) + 授权层 (ACL namespace 隔离) |
| 7 | **协议枚举编译期对齐** | Rust / Java / MySQL 三端协议 code 严格一致 (ADR-08)，预防性设计 |
| 8 | **C4 + DDD 视角架构文档** | 限界上下文、聚合根、容量规划均完备，12+ 份文档覆盖全栈 |
| 9 | **ConfigPushService 按协议差异区分配置下发** | Modbus / DLT645 / MQTT / OPC UA 等不同协议的 data_points 做差异化处理 |
| 10 | **采集端 LWT (Last Will Testament)** | 采集器断连自动上报离线状态 |
| 11 | **安全体系完整** | JWT 双 Token + RBAC 三角色 + @AuditLog 审计切面 + 登录限流 |

## 三、问题清单

### 3.1 严重问题 (P0 — 阻塞联调)

#### ❌ BUG-001: ConfigPushService 协议路由逻辑错误

| 字段 | 内容 |
|------|------|
| **位置** | `ConfigPushService.java` 第 163-204 行，`buildBusConfig()` 方法 |
| **严重程度** | P0 — 阻塞所有 TCP 协议设备部署 |
| **问题** | switch 匹配 `ctx.protocol.getCode()` (即 "MODBUS_TCP", "BACNET_IP" 等)，但分支只处理了 "MQTT" 和 "tcp" 两个字面量。没有协议 code 等于 "tcp"，因此**所有 TCP 协议设备（Modbus TCP、BACnet/IP、OPC UA、IEC 104、DNP3、S7、FINS、EtherNet/IP、Mitsubishi MC、SNMP）全部落入 default 分支**，被错误分配了串口 bus 配置 |
| **影响** | 所有 TCP 协议设备下发到采集端的配置中，bus 结构是串口参数而非 TCP 参数，采集端无法建立 TCP 连接 |
| **修复方向** | 应使用 `ctx.protocol.getBusType()` 或按协议编码集合匹配，正确区分 serial / tcp / mqtt 三种总线类型 |

```
当前错误逻辑:
  case "MQTT" → 构建 MQTT bus
  case "tcp"  → 构建 TCP bus      ← 这个词不可能匹配任何协议code
  default     → 构建 serial bus    ← 所有TCP协议都落入这里

正确逻辑:
  case "MQTT"                    → 构建 MQTT bus
  case "MODBUS_TCP","BACNET_IP"... → 构建 TCP bus
  case "MODBUS_RTU","DLT645"...   → 构建 serial bus
```

#### ❌ BUG-002: Aggregator 和 AlarmEngine 未集成到调度器主循环

| 字段 | 内容 |
|------|------|
| **位置** | `collector-scheduler/src/lib.rs` 的 `run()` 方法 |
| **严重程度** | P0 — 双通道架构中通道 B 完全未运转 |
| **问题** | 架构文档描述的双通道上报+边缘告警，在代码中**只有单通道在运行**。调度器每次 poll 仅执行：driver.collect() → 构造 LatestReading → uploader.publish()。完全未调用 Aggregator.push() 和 AlarmEngine.check() |
| **影响** | (1) 降采样聚合不工作，历史数据无法按窗口聚合入库 (2) 告警功能完全失效 — 边侧不判断，服务端也不判断 |

#### ❌ BUG-003: Aggregator 内部硬编码 bug

| 字段 | 内容 |
|------|------|
| **位置** | `collector-reporter/src/lib.rs` 第 71-79 行 |
| **严重程度** | P0 — 即使集成也输出不了有效数据 |
| **问题** | `device_id: 0`（硬编码），`window_start: ""`（空字符串），`window_end: ""`（空字符串） |
| **影响** | 聚合后的 `AggregatedReading` 无法关联到具体设备，窗口时间信息丢失 |

### 3.2 重要问题 (P1 — 生产就绪前必修)

#### ⚠️ BUG-004: 实时数据直接写入 MySQL（与架构设计矛盾）

| 字段 | 内容 |
|------|------|
| **位置** | `MqttSubscriberService.java` |
| **严重程度** | P1 — 架构破坏 |
| **设计约定** | 架构报告第 4.2 节明确：通道 A (实时) → 服务端内存 → WebSocket → 前端，**不入库** |
| **实际行为** | 每条 `LatestReading` 都执行 `readingMapper.insert()` 直接写入 MySQL |
| **影响** | 高采集频率下（每秒轮询），MySQL 承受不必要的写入峰值。与双通道架构设计初衷背道而驰 |

#### ⚠️ GAP-001: WebSocket 服务端 + 前端完全缺失

| 字段 | 内容 |
|------|------|
| **位置** | 全项目搜索 WebSocket 相关代码 |
| **严重程度** | P1 |
| **设计约定** | 架构图显示"服务端内存 → WebSocket → 前端 (<100ms)"实时链路 |
| **实际状态** | 服务端代码中完全不存在 WebSocket 端点；前端无 WebSocket 客户端连接代码 |
| **影响** | 前端获取实时数据只能通过 API 轮询，延迟高、资源浪费 |

#### ❌ BUG-005: 设备通讯状态管理完全缺失

| 字段 | 内容 |
|------|------|
| **位置** | `collector-scheduler/src/lib.rs` + `collector-model/src/lib.rs` |
| **严重程度** | P1 — 无法感知设备在线/离线/故障 |
| **现象** | Device 结构体 `collector-model/src/lib.rs:163-174` 没有任何运行时状态字段（无 online、无 last_success_at、无 consecutive_failures）。调度器每轮 poll 仅判断 Ok/Err，不做失败计数、不做超时保护、不做状态变更上报 |
| **三大缺口** | (1) `spawn_blocking` 未包装 `tokio::time::timeout`，某设备串口阻塞则调度器永久卡住 (2) 无连续失败计数/backoff 策略，失败只是打一行 warn 继续循环 (3) 不区分错误类型——串口打不开、TCP超时、Modbus异常码、CRC校验错全部混在一起 |
| **附加问题** | Modbus 驱动中异常码被 `.unwrap_or_default()` 静默吞掉 — 设备返回了异常码，调度器却看到"成功但 0 条数据"，完全不知道出了错 |
| **影响** | 采集器离线后挂载设备不会级联标记离线；设备通讯质量全然不知；运维无法判断是偶尔闪断还是设备已死 |

#### ⚠️ GAP-002: 采集端日志体系粗糙

| 字段 | 内容 |
|------|------|
| **位置** | 全局 — tracing 配置 + 16 个驱动子模块 |
| **严重程度** | P1 — 现场故障无法诊断 |
| **现象** | (1) 16 个驱动子模块内部**零日志** — 调度器看到的是黑盒结果，串口握手耗时、TCP 重连次数、设备异常码全部不知 (2) 全项目无 `elapsed_ms` 耗时字段 — 通讯变慢是设备故障前兆，当前完全感知不到 (3) 全项目无 `error_type` 分类 — 超时/断连/校验错/协议异常混在一起 (4) 日志仅 stdout 文本输出 — 无文件持久化、无 JSON 结构化，采集器现场出问题看不到历史日志 (5) 日志级别硬编码 — 不支持 `RUST_LOG` 环境变量动态调级 |
| **verb** | 日志仅"记录"不"触发"——理想状态是：通讯超时 3 次连续 → 日志 WARN → 设备标记 offline → MQTT 发 status/offline → 服务端感知 → 前端红色。当前整条链路不存在 |

#### ⚠️ GAP-003: 前端 RefreshToken 刷新机制缺失

| 字段 | 内容 |
|------|------|
| **严重程度** | P1 |
| **问题** | JwtTokenProvider 支持生成 refreshToken，但前端 `stores/auth.ts` 只管理 accessToken。refreshToken 的存储、静默刷新、黑名单均未实现 |
| **影响** | 用户 30 分钟后被迫重新登录 |

#### ⚠️ GAP-004: MySQL 分区自动管理任务缺失

| 字段 | 内容 |
|------|------|
| **严重程度** | P1 |
| **问题** | 分区是静态预创建的（仅到 p202608），缺少：1) 每月自动建下月分区 2) 按 `data_retention_days` 自动 DROP 过期分区 |
| **影响** | 2026年8月后数据无法写入，历史数据无限增长 |

#### ⚠️ GAP-005: 采集端使用 JSON 文件而非 SQLite

| 字段 | 内容 |
|------|------|
| **严重程度** | P1 |
| **设计约定** | 采集端设计文档要求 SQLite 本地缓存 |
| **实际状态** | 使用 NDJSON 文件 (`devices.json`)，原因是编译环境缺少 C 编译器无法编译 rusqlite |
| **影响** | 高频采集场景下文件追加的性能和并发安全性存疑 |

#### ⚠️ GAP-006: 协议判断列表前后端各自维护一份硬编码

| 字段 | 内容 |
|------|------|
| **位置** | 前端 `device/index.vue` 的 `getProtocolBusType()` + 后端 `ConfigPushService.buildBusConfig()` |
| **严重程度** | P1 — 必炸型不一致隐患 |
| **问题** | 前后端各自维护"哪些协议是 TCP"的硬编码列表，互相不知道对方。加新协议时一处漏改则前端显示正确定后端推送错误 |
| **修复** | 后端 `/api/v1/protocols` 返回数据应带 `busType` 字段，前端直接读取不自己枚举 |

## 四、数据流审计

### 4.1 设计文档中的理想数据流

```
采集端 ──LatestReading (QoS0)──→ MQTT Broker ──→ 服务端内存 ──WebSocket──→ 前端 (<100ms)
采集端 ──Aggregator.window──→ AggregatedReading (QoS1) ──→ MQTT ──→ MySQL ──API──→ 前端
采集端 ──AlarmEngine.check()──→ AlarmEvent (QoS1) ──→ MQTT ──→ MySQL ──→ 通知
采集端 ──通讯状态变更──→ status topic ──→ 服务端设备状态更新 ──→ 前端
```

### 4.2 代码中实际运行的数据流

```
采集端 ──LatestReading (QoS0)──→ MQTT Broker ──→ 服务端 MqttSubscriber ──直接写 MySQL──→ 前端 API 轮询
采集端 ──Aggregator: 未调用──→ (不工作)
采集端 ──AlarmEngine: 未调用──→ (不工作)
采集端 ──设备通讯状态: 无跟踪──→ (不工作)
```

### 4.3 差距分析

| 数据流环节 | 设计 | 实际 | 差距 |
|-----------|------|------|------|
| 实时通道推送 | MQTT → 服务端内存 → WebSocket | MQTT → 直接写 MySQL | **严重不一致** |
| 历史聚合通道 | Aggregator 窗口聚合后 MQTT 上报 | Aggregator 未被调用 | **完全缺失** |
| 边缘告警 | AlarmEngine 本地判断后 MQTT 上报 | AlarmEngine 未被调用 | **完全缺失** |
| 设备通讯状态 | poll 成功/失败 → 设备状态变更 → status 上报 | 无跟踪、无上报 | **完全缺失** |
| WebSocket | 服务端推送实时数据到前端 | 不存在 | **完全缺失** |
| 前端实时获取 | WebSocket 订阅 | API 轮询 | **降级方案替代了主方案** |

## 九、综合评判

### 架构设计评分: A (优秀)

项目在架构设计层面表现优秀。双通道上报、边缘计算、降采样、协议编译期对齐、MQTT 三层安全等设计决策均体现了成熟的工业 IoT 思维。18 种协议全部帧级驱动实现更是在行业中极为罕见。12+ 份文档构成完整体系。

### 代码实现评分: C+ (及格但差距明显)

代码质量本身尚可，但存在 **"设计-实现鸿沟"** + **"可观测性真空"**：

- **鸿沟**：架构文档描述的三通道高速公路，代码里**只有一条在跑**。告警引擎和降采样聚合器虽然代码写了，但从未被调度器主循环调用
- **真空**：设备通讯质量、在线状态、错误分类、通讯耗时——这些工业 IoT 的核心可观测性指标，全项目完全缺失
- **隐患**：ConfigPushService 协议路由 bug 影响面极大，前端交互体验存在多处粗糙点

### 核心矛盾

> **设计文档描述的理想架构 vs 代码中实际运行的单通道简化版**  
> **工业 IoT 对"可观测性"的高要求 vs 代码中通讯状态/日志体系的近乎空白**

修复 P0（2.5 天）打通三通道 + P1 的通讯状态管理和日志增强（3 天），再补数据流闭环（4.5 天），项目就能从"设计优秀但实现残缺"变成"设计优秀且实现完整且状态可观测"。
