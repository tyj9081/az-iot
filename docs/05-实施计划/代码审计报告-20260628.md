## 🔴 P0 — 致命: 必须立即修复

### R-P0-001: Aggregator + AlarmEngine 永不清理 (内存泄漏)

**文件:** `collector-scheduler/src/lib.rs:69-71`
**影响:** 设备动态增删时 HashMap 无限增长 → OOM

```rust
aggregators: Arc<tokio::sync::Mutex<HashMap<i64, Aggregator>>>,  // 只增不删
alarm_engines: Arc<tokio::sync::Mutex<HashMap<i64, AlarmEngine>>>, // 只增不删
```

### R-P0-002: Modbus 异常码被静默转为数值 0.0 → 污染双通道数据

**文件:** `collector-driver/src/modbus.rs:41-87`
**影响:** 设备异常返回的 0.0 会流入 Aggregator 窗口(参与avg/max/min计算) 和 AlarmEngine(可能触发虚假下限告警)

```rust
Err(exception) => {
    tracing::warn!("Modbus exception ...");
    Vec::new()  // → 下游 decode 返回 0.0，被视为"正常读数"
}
```

### R-P0-003: HTTP fallback URL 主机名损坏

**文件:** `collector-uploader/src/lib.rs:366-368`
**影响:** `ws://ws-gateway.example.com:8080/ws/collector` 被替换为 `http://http-gateway.example.com:8080/ws/collector`，fallback 永远不可用

```rust
let http_url = self.fallback_config.ws_url
    .replace("ws://", "http://")     // 全局替换 → 主机名也被改!
```

## 架构建议

1. **采集端: ProtocolDriver::collect 改为 async fn** — 当前同步 trait + block_on 在 spawn_blocking 线程池运行异步 I/O，既浪费线程又有池耗尽风险
2. **服务端: MQTT 消息处理加死信队列 + 批量写入** — 当前单条 insert + 无重试，高频场景下 DB 压力极大
3. **前端: 建立 TypeScript Domain Model 层** — 消除 60+ 处 `any`，用 Zod/io-ts 做 API 响应运行时校验
