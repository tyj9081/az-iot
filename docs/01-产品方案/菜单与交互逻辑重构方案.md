# AZ-IOT 菜单与交互逻辑重构方案 v2

> **核心原则：操作保持线性，每一步的完成自然引出下一步。**  
> **关键修正：采集器是单例（MQTT 自动订阅），只需配置采集周期，不需要独立管理页。**

---

## 一、现状诊断

### 1.1 当前菜单

```
工作台          /dashboard         ← 第一个，但没数据时是空的
设备模板
  品牌管理       /manufacturer
  型号管理       /device-model
采集器管理       /collector        ← 独立页面，但实际只有 1 台，配置项极少
设备管理         /device
```

### 1.2 三个核心问题

| # | 问题 | 影响 |
|---|------|------|
| **1** | 工作台排第一，但没设备时完全空 | 用户进来不知道从哪开始 |
| **2** | 采集器是单例（一台 BC-U101，MQTT 自动订阅），却独占一个管理页 | 过度设计，用户困惑"这页面要干嘛" |
| **3** | 型号创建后要跳到另一个子页面录入点表，流程割裂 | 应该在一个页面内完成：选品牌 → 填型号 → 导点表 |

### 1.3 实际线性依赖链（精简版）

```
品牌 ──→ 型号(含点表导入) ──→ 设备(含采集器设置) ──→ 工作台
 ①          ②                   ③                   ④
```

## 二、新菜单结构

```
┌─────────────────────────┐
│ 📋 设备接入              │
│   ① 品牌管理             │  ← 先定义厂商
│   ② 型号管理             │  ← 创建型号 + 导入点表（文件上传 / JSON 粘贴）
│   ③ 设备管理             │  ← 创建物理设备 + 采集器设置（采集周期等）
│                         │
│ 📊 运行监控              │
│   ④ 工作台               │  ← 查看实时数据
└─────────────────────────┘
```

**只有 4 个菜单项**，线性推进，无冗余。

### 2.2 和上一版的差异

| 变更 | v1 方案 | v2 方案 | 原因 |
|------|---------|---------|------|
| 采集器管理 | 独立页面 ③ | **删除**，合并到设备管理页 | 单例硬件，配置项极少，不需要独立页 |
| 菜单数 | 5 项 | **4 项** | 更简洁 |
| 型号管理 | CRUD + 跳转子页面录点表 | **内嵌点表导入**（文件上传 + JSON 粘贴） | 一个页面完成全流程 |

**交互逻辑**：
1. 默认选中「稍后配置」，弹窗保持当前简洁形态

3. 两个 tab：**上传文件**（支持 .json / .csv）和 **JSON 粘贴**（textarea）
4. 粘贴或上传后，下方实时预览解析出的点位列表
5. 点击「创建」→ 先创建型号 → 再批量导入点表 → 成功

**型号列表页改动**：
- 型号表格的「点表管理」按钮仍然保留（用于后续查看/修改）
- 未配置点表的型号行显示 ⚠️ 黄色标记
- 底部加「下一步 → 设备管理」引导

#### 设备与串口的对应关系

**数据链路**（三层外键）：

```
DevDevice ──serial_port_id──> DevSerialPort ──collector_id──> DevCollector
  设备                          串口                           采集器
                              ├─ port_name: COM3
                              ├─ bus_param: {"baud":9600,"data_bits":8,...}
                              └─ port_type: device
```

**创建设备时的操作逻辑**：
1. 系统只有一台采集器，自动选中（无需用户选择）
2. 加载该采集器下所有 `port_type = 'device'` 的串口（COM3~COM10）
3. 用户选择串口 → 选择型号 → 填写从站地址 → 创建
4. 一个串口可以挂多个设备（靠从站地址区分）

**波特率是关键约束**：不同波特率的设备必须挂在**不同 COM 口**上。例如 TH-802C 用 9600，另一个 DLT645 电表固定 2400，它们不能共用同一个 COM3，需要分别挂 COM3 和 COM4，各自设好波特率。

**MQTT 配置下发链路**：
```
创建设备 → 查 serial_port → 取 port_name + bus_param → 
拼 JSON → MQTT Topic "neuron/{clientId}/config/delta" → 
Rust 采集端解析 → Modbus 驱动用 bus_param.baud 打开串口开始轮询
```

## 五、完整操作流程（用户视角）

```
第1步：品牌管理
  → 录入"威胜""北京昆仑海岸"等厂商
  → 点「下一步」或侧栏进入型号管理

第2步：型号管理
  → 点「新增型号」
  → 选择品牌 + 协议，填写编码/名称
  → 选「立即导入」
  → 粘贴 JSON 或上传文件
  → 预览确认 6 条点位
  → 点「创建」完成
  → 点「下一步」或侧栏进入设备管理

第3步：设备管理
  → 展开「采集器设置」确认采集周期
  → 点「新增设备」
  → 选择采集器、串口、型号、填写从站地址
  → 创建完成
  → 点「下一步」或侧栏进入工作台

第4步：工作台
  → 看到设备在线、数据在采集
  → 日常监控
```

## 八、附录：关键数据结构速查

### bus_param JSON 结构

```json
{
  "baud": 9600,
  "data_bits": 8,
  "stop_bits": 1,
  "parity": "none"
}
```

### 串口默认分配（创建采集器时自动生成）

| 串口 | port_type | 用途 | 默认 bus_param |
|------|-----------|------|----------------|
| COM1 | sms_modem | 短信猫 | 9600-8N1 |
| COM2 | io_board | IO 板卡 | 9600-8N1 |
| COM3~COM10 | device | 普通设备 | 9600-8N1 |

### 设备-串口关系约束

| 约束 | 说明 |
|------|------|
| 一个设备 → 一个串口 | `dev_device.serial_port_id` 必填 |
| 一个串口 → 多个设备 | 靠从站地址(slave_addr)区分 |
| 同串口设备 → 同波特率 | 不同波特率设备必须挂在不同 COM 口 |
