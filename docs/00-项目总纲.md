# AZ-IOT 开发总纲

> 2026-06-25 | v1.0.0

---

## 一、项目定位

面向工业园区/楼宇的能源管理 IoT 平台。采集端从 BC-U101 工控机获取设备数据 → 服务端存储加工 → 前端展示监控。

**架构**：`Vue3 前端 → Java 服务端 → Rust 采集端 → BC-U101 → 现场设备`

**数据通道**：HTTP REST (用户交互) + WebSocket (实时推送) + MQTT (采集上下行)

---

## 二、功能范围 (MVP)

### 做什么

1. 品牌/型号/协议字典管理
2. 设备型号 + 寄存器点表 CRUD，支持 Excel 批量导入
3. BC-U101 采集器注册 + 串口管理 (COM1-COM10)，创建时自动生成默认串口
4. 设备绑定型号/串口/从站地址 → MQTT 配置下发 → 采集端自动纳管
5. 双通道数据上报：通道A(实时,不进DB) + 通道B(降采样聚合,进MySQL)
6. 告警边缘判断（采集端本地执行，断网仍可联动 DO 继电器声光报警）
7. 采集数据展示：通用渲染器按 data_type 自动选组件
8. Dashboard 概览页 + 设备详情页 + 历史趋势图
9. JWT 双 Token 认证 (30min access + 7d refresh)
10. 三角色 RBAC (admin/operator/viewer) + 审计日志

### 不做什么

能耗分析、线损、BA楼控、报表、生产管理、光伏/空调 —— 后置至 V2

## 三、数据模型

```
dev_protocol ──┐
               │ 1:N
dev_manufacturer ──1:N──→ dev_device_model ──1:N──→ dev_register_map
                                     │
                                     │ 1:N
                                     ↓
                               dev_device ──1:N──→ dev_device_reading
                                     │            (含 register_id FK)
                                     │ N:1
dev_collector ──1:N──→ dev_serial_port ──┘

sys: user, role, permission, user_role, role_permission, audit_log, config
```

### 核心表

| 表 | 关键字段 | 说明 |
|----|---------|------|
| dev_protocol | code, name, bus_type | 协议字典 (ModbusRTU, DL_T645...) |
| dev_manufacturer | code, name, country | 设备品牌 (威胜, 正泰...) |
| dev_device_model | manufacturer_id FK, protocol_id FK, code, name, collect_interval_sec | 设备型号 |
| dev_register_map | model_id FK CASCADE, sensor_code, sensor_name, register_address, register_count, data_type, byte_order, func_code, coefficient, offset_val, unit, rw, sort_order | 寄存器点表 |
| dev_collector | code, name, type, mqtt_client_id, ip_address, status | 采集器 (BC-U101) |
| dev_serial_port | collector_id FK, port_name, bus_type, bus_param JSON, port_type | 串口配置 |
| dev_device | serial_port_id FK, model_id FK, code, name, slave_addr, collect_interval_sec | 终端设备 |
| dev_device_reading | device_id FK, register_id FK, sensor_code, value, avg, max, min, sample_count, quality, read_at | 时序读数 (按月分区) |

## 四、架构决策记录 (ADR)

### ADR-01 — 读数表按月分区

```sql
PARTITION BY RANGE (TO_DAYS(read_at))
```
- 每月 1 号定时任务创建下月分区
- 超期数据 DROP PARTITION (秒级完成, 零锁)

### ADR-02 — 数据保留两层

| 层 | 默认 | 配置 | 清理 |
|----|-----|------|------|
| 采集端 SQLite | 7天 (循环覆盖) | config.toml | 定时 DELETE |
| 服务端 MySQL | 90天 | sys_config API | DROP 分区 |
| 审计日志 | 180天 | sys_config API | 批量 DELETE |

### ADR-03 — 双通道上报

```
通道A (实时): latest (QoS0) → 服务端内存 → WebSocket → 前端 (<100ms)
通道B (历史): 降采样聚合 → reading (QoS1) → MySQL 分区表 → API
```
- 通道A 不进 DB，纯内存缓存 + WebSocket 广播
- 通道B 采集端本地 Aggregator 攒够窗口 → avg/max/min → 上报

### ADR-04 — 告警边缘判断

- 采集端每次 poll 后本地阈值判断 (非服务端远程判断)
- 断网状态下 DO 继电器仍可联动声光报警
- 服务端只负责告警记录存储 + 通知分发

### ADR-05 — 采集端降采样

| 采集周期 | 窗口 | MySQL 日写入 (47台×20点) |
|---------|------|------------------------|
| 1秒 | 60秒 | 27万 |
| 5秒 | 180秒 | 9万 |
| 10秒 | 300秒 | 5.4万 |
| ≥30秒 | 不聚合 | ≤28万 |

上报荷载: `{sensor_code, avg, max, min, sample_count, window_start, window_end}`

## 五、开发规范

### S01 — API 契约

- 统一响应: `{code, message, data, timestamp}`
  - code: 0=成功, 400/401/403/404/500
- 分页: `{records, total, page, pageSize}`, page 从 1 开始
- JSON 字段 camelCase, 日期 `yyyy-MM-dd HH:mm:ss`
- 入参出参使用强类型 DTO/VO, **禁止 Map<String,Object>**
- Controller 类级别: `@RequestMapping("/api/v1/xxx")`
- 统一异常处理: `@RestControllerAdvice`

### S02 — 数据库

- 表前缀: 业务 `dev_`, 系统 `sys_`
- 主键 `id BIGINT AUTO_INCREMENT`
- 审计字段: `created_at`, `updated_at ON UPDATE CURRENT_TIMESTAMP`
- **禁止 ENUM** (用 VARCHAR + 注释说明)  
- **禁止关键字段 NULL** (NOT NULL + DEFAULT)
- 外键字段必有索引, 索引命名 `idx_{表简写}_{列名}`
- Flyway 版本号: `V{序号}__{描述}.sql` (双下划线), **禁止修改已执行文件**
- 逻辑删除: `is_deleted TINYINT DEFAULT 0`

### S03 — MQTT 消息

- Topic: `neuron/{collector_mqtt_client_id}/{type}`
- type: reading, latest, heartbeat, alarm, config/delta, config/full, config/ack, cmd, cmd/result
- 荷载 JSON, **禁止包含 Java 包名或 Rust 模块路径**
- 通道A (latest): QoS 0, fire-and-forget
- 通道B (reading): QoS 1, 批量 ≤500 条/消息
- quality 定义: 0=正常, 1=超时, 2=异常, 3=传感器故障
- value 字段为已解析工程值, 禁止传原始字节

### S04 — Java 代码

- 构造器注入: `@RequiredArgsConstructor`, **禁止 @Autowired 字段注入**
- 分层: Controller → Service → Mapper, **禁止 Controller 直接调 Mapper**
- Service 返回 VO, **禁止 Controller 暴露 Entity**
- 包结构按业务域: `entity.device/collector/system`
- 异常: `@RestControllerAdvice` 全局处理, **禁止 Controller 内 try-catch 吞异常**
- 日志: `@Slf4j`, info=正常, warn=可恢复, error=需介入
- **禁止 e.printStackTrace()**
- 循环依赖视为模块划分问题, 必须解决

### S05 — Rust 代码

- `cargo clippy` 零警告, `cargo fmt` 统一格式
- **禁止 `unwrap()`** 在非测试代码 (用 `?` 或 `expect("why")`)
- **禁止吞错误** `let _ = ...` 
- **禁止 `println!`**, 用 `tracing` (info/warn/error)
- Protocol trait: `async fn collect` 内部用 `spawn_blocking` 包裹同步 IO
- 错误处理: `anyhow::Result` 顶层, `thiserror` 结构化错误
- 序列化用 `serde`, 字段名 snake_case

### S06 — Vue 3 代码

- `<script setup lang="ts">`, Composition API, Composables 复用
- Props/Emits 声明类型, **禁止 `any`** (用 `unknown`+类型守卫)
- 组件命名: PascalCase (文件), kebab-case (views 目录)
- API 调用: 必须经 `src/api/` 封装, **禁止 .vue 内直接 axios**
- 状态: 全局→Pinia, 页面级→ref/reactive
- Element Plus 中文 locale
- TypeScript strict mode

### S07 — 测试规范

- **业务代码严禁任何 mock, 所有测试走真实实现**
- Java: `@SpringBootTest + H2` (Flyway 自动建表), `@Transactional` 自动回滚
- Rust: 已知字节→已知数值, 每种 data_type × 4 byte_order 全覆盖
- 提交前置检查: `mvn verify` / `cargo test && cargo clippy` / `pnpm build`
- **禁止** `mock(Service.class)` / `when().thenReturn()` / 空壳测试

```
├── README.md

│   └── 开发总纲.md

│   ├── collector-storage/          # SQLite 缓存, 断网续传
│   ├── collector-telemetry/        # 心跳, metrics
│   └── collector-bin/              # main, config.toml
└── neuron-web/                     # Vue 3 SPA
    ├── src/api/                    # 请求层 (auth, manufacturer, device...)
    ├── src/views/                  # login, dashboard, device, system...
    ├── src/components/             # DataTable, ValueCard, StatusLight, TrendChart
    ├── src/stores/                 # Pinia (auth)
    ├── src/types/                  # TS 类型 (对齐 Java DTO)
    └── src/router/                 # 路由配置
```
